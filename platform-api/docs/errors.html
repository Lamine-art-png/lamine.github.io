Errors

The API uses conventional HTTP status codes and returns a typed, machine-readable error body.

Error format

Every error response has the same shape. Branch on error.code, show error.message, and use error.request_id when contacting support.

422 · application/json
{
  "error": {
    "type": "invalid_request_error",
    "code": "parameter_invalid",
    "message": "horizon_days must be between 1 and 14.",
    "param": "horizon_days",
    "request_id": "req_9F2xa7Qh"
  }
}

HTTP status codes

StatusMeaning
200 / 201Success.
400Malformed request (e.g. invalid JSON).
401Missing or invalid API key.
403Authenticated but not permitted.
404Resource does not exist.
409Conflict (e.g. idempotency key reuse with a different body).
422Validation failed on an otherwise well-formed request.
429Rate limited — see Rate limits.
500 / 503Something went wrong on our side. Retry with backoff.

Error codes

codeWhen
authentication_requiredNo credentials were supplied.
invalid_api_keyThe key is unknown or revoked.
parameter_invalidA parameter failed validation. See param.
resource_missingThe referenced id does not exist.
rate_limitedToo many requests in the current window.
api_errorInternal error. Safe to retry idempotent calls.

Retries

Retry 429 and 5xx responses with exponential backoff and jitter. Use an idempotency key on writes so a retried request is never applied twice.
