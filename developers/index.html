Use one organization-scoped API for field records, source ingestion, observations, recommendations, reports, provider readiness, durable webhooks, usage, and safe request logs.

Quickstart

Create a test project, service account, and scoped test key in the Developer Console. Plaintext keys are shown once.

curl https://api.agroai-pilot.com/v1/platform/me \
  -H "Authorization: Bearer $AGROAI_API_KEY" \
  -H "X-Request-Id: quickstart-001"
from agroai_platform import AgroAI

client = AgroAI(api_key=os.environ["AGROAI_API_KEY"])
for field in client.fields.list():
    print(field.id, field.name)
import { AgroAI } from "@agro-ai/platform";

const client = new AgroAI({ apiKey: process.env.AGROAI_API_KEY! });
for await (const field of client.fields.list()) {
  console.log(field.id, field.name);
}

Authentication

Send a Platform API key as a Bearer token. Portal JWTs are not API keys. Test keys begin with agro_test_; live keys begin with agro_live_.

Projects and environments

Projects are structurally test or live. Test projects use deterministic synthetic data. Live projects require separate live-access approval, an eligible API subscription or contract, and production readiness. A test project is never promoted into a live project.

Service accounts and API keys

Service accounts are machine identities with bounded scopes and optional workspace, provider, and resource restrictions. Keys may narrow those permissions but cannot widen them. Rotate with bounded overlap; revoke or disable immediately when no longer needed.

Scopes and CIDR restrictions

Common scopes include fields:read, fields:write, sources:read, observations:write, recommendations:write, reports:read, webhooks:write, and usage:read. CIDR lists accept validated IPv4 and IPv6 networks. A CIDR-bound key fails closed unless the Cloudflare edge provides an authenticated client address.

Fields

Create, retrieve, list, update safe metadata, and archive fields. Boundaries accept GeoJSON Polygon or MultiPolygon values.

curl -X POST https://api.agroai-pilot.com/v1/platform/fields \
  -H "Authorization: Bearer $AGROAI_API_KEY" \
  -H "Idempotency-Key: field-create-001" \
  -H "Content-Type: application/json" \
  -d '{"name":"North Block","crop":"almond","area_hectares":24.6}'

Sources and ingestion

Register structured sources or initiate a direct R2 upload with an expected SHA-256 checksum. Uploaded objects remain in an organization and project namespace. Structured observation batches create durable Queue-backed jobs.

Observations

List by field, type, and time range. Every response includes provenance and quality flags. Batch payloads are bounded and processed as one logical metered operation.

Recommendations

Recommendation computation is asynchronous. POST a request with an idempotency key, poll the returned job, then retrieve the recommendation with evidence and provenance.

Reports

Report generation is asynchronous. Report metadata never exposes raw object-storage paths; content is returned through an authorized download route.

Connectors and jobs

The provider catalog reports customer-available capabilities and truthful readiness. Synchronization is restricted by key, enrollment, provider, and resource policy. Job retries are allowed only for eligible terminal failures and reuse the same durable outbox custody.

Webhooks

Events include an event ID, version, and timestamp. Deliveries use a timestamped HMAC-SHA256 signature, bounded network behavior, exponential backoff, and final failure state. Signing secrets are shown once and retained encrypted for the delivery worker.

AgroAI-Signature: t=1784448000,v1=<hex-hmac>
AgroAI-Event-Id: <event-id>
AgroAI-Event-Version: 2026-07-01

Pagination and filtering

Collection routes use bounded limit values and opaque cursor tokens. Responses contain items, next_cursor, and has_more. Do not construct cursors manually.

Idempotency

Every logical write accepts or requires Idempotency-Key. A completed match replays the stored response, a concurrent match returns operation_in_progress, and payload reuse with different content returns 409. Internal retries are not metered again.

Errors and request IDs

Errors have code, type, message, and request_id. Send a bounded X-Request-Id or use the returned value when contacting support.

StatusMeaning
400/422Invalid request or schema
401Invalid key or trusted-client-address denial
403Scope, provider, resource, enrollment, or environment denial
409Conflict or operation already in progress
429Rate or credit limit

Rate limits and API credits

Rate-limit headers are RateLimit-Limit, RateLimit-Remaining, and RateLimit-Reset. The AGRO-AI ledger authorizes credits synchronously through reservation, reconciliation, and failure release. Stripe meter export is asynchronous settlement only.

Billing

API plans and subscriptions are separate from Enterprise Portal plans. Checkout is server-created from the active versioned catalog; browsers never submit price IDs or amounts. Billing details, invoices, and payment methods use Stripe Customer Portal.

Deterministic sandbox

Test projects include deterministic synthetic organizations, farms, fields, zones, crops, seasons, irrigation systems, observations, events, anomalies, recommendations, reports, jobs, and webhook events. The fixture is resettable and never includes customer data, provider credentials, real charges, live webhooks, or physical execution.

SDKs

The unpublished Python and TypeScript packages provide typed errors, request IDs, rate-limit state, safe retries, pagination, uploads, polling, usage access, environment selection, and server-side webhook verification. Browser runtimes must never receive an API key.

Changelog, versioning, and deprecation

The curated OpenAPI document is the contract source. Breaking changes require a versioned route or reviewed deprecation window. SDK compatibility and OpenAPI diff checks run in CI.

Security

Use the minimum scopes and restrictions, rotate keys, validate webhook timestamps, deduplicate event IDs, and keep keys in server secret stores. Never put API keys in browser bundles or Vite variables.

Support and status

The Developer Console links support requests to safe request IDs, projects, jobs, delivery attempts, and invoices. The status foundation reports components and incidents without uptime or SLA claims.

Strategic partner integrations

Custom provider work uses a separate technical and commercial review, integration dossier, encrypted credential vault, contract blockers, custom rate card, and explicit sandbox/read/write/production readiness. Provider credentials never belong in application or dossier JSON.
